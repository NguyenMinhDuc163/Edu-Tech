## Project overview

EdTech is a Flutter/Dart learning app with Firebase/Google sign-in, Dio-based backend APIs, BLoC state management, Easy Localization, responsive layouts, local/secure storage, course browsing, quizzes, chatbot, payments, profile, and purchased-course flows. The app targets Android, iOS, Web, and desktop platforms generated by Flutter.

## Common commands

```bash
# Install dependencies
flutter pub get

# Static analysis/lints
flutter analyze

# Run all tests
flutter test

# Run a single test file
flutter test test/icon_path_test.dart

# Run one test by name
flutter test test/icon_path_test.dart --plain-name "icon_path assets test"

# Run the app
flutter run

flutter run -d ios

# Build release artifacts

flutter build web --release -t lib/main_preview.dart
```

The project uses `flutter_lints` via `analysis_options.yaml`; `flutter analyze` is the canonical lint/static-analysis command.

## Environment and generated configuration

- Runtime API configuration is generated with `envied` from `env/.env.dev`; `lib/core/constants/environment_files.dart` sets `pathToEnv = 'env/.env.dev'`.
- `lib/core/constants/api_path.dart` declares `@Envied(path: pathToEnv, requireEnvFile: true)` and exposes obfuscated `BASE_URL` and `REGISTER_ID` plus endpoint constants.
- `lib/core/constants/api_path.g.dart` is generated. Re-run build_runner when env fields change.
- Firebase options live in `lib/firebase_options.dart`, but `Firebase.initializeApp(...)` is currently commented out in `lib/main.dart`; verify initialization before adding Firebase-dependent behavior.

## App startup and composition

- Entry point: `lib/main.dart` initializes Flutter bindings, installs a global HTTP override that accepts bad certificates, sets `AppBlocObserver`, ensures Easy Localization, initializes `AuthService` and `UserService`, then wraps `App` in `EasyLocalization` for `en_US` and `vi_VN` translations from `assets/translations`.
- Root widget: `lib/modules/app.dart` creates global providers for sign-in, theme, and locale, configures `MaterialApp`, responsive breakpoints, routes, `NavigationService.navigatorKey`, and `NavigationService.routeObserver`.
- `App` implements `AppAuthenticationBindingObserver`; auth failure/token expiration events clear tokens and navigate to `LoginScreen` with `pushNamedAndRemoveUntil`.
- `lib/init.dart` is a convenience export barrel for Flutter/material, common theme constants, asset paths, and screen templates.

## Architecture map

- `lib/modules/` contains feature areas. Most features are organized as `screen/`, `bloc/`, `repository/`, `model/`, and `widgets/` or `widget/` subfolders.
  - Existing feature areas include auth, home, course, assessment, message/chatbot, payment, profile, purchased_courses, reviews, ranking, dashboard, and account.
- `lib/core/` contains app-wide primitives: route definitions, navigation service, authentication binding, constants, theme system, shared widgets, error handling, debouncing, and general utilities.
- `lib/data/` contains cross-feature infrastructure: `ApiClient`, Dio interceptors, secure storage, common request/response models, connectivity, auth, user, and error-dialog services.
- `lib/common/` contains shared app status/event models and reusable common widgets.
- `lib/utils/` contains helpers/extensions, preferences, and UI utility extensions.
- `assets/icons`, `assets/images`, and `assets/translations` are registered in `pubspec.yaml`; tests assert that generated asset path constants point to existing files.

## Routing and dependency injection pattern

- Central routing lives in `lib/core/routes/routers.dart`.
- `Routers.generateRoute` switches on each screen's static `routeName`, records `GlobalUtils.ROUTES`, and builds feature-specific dependency graphs inline.
- Feature routes typically wrap screens with `RepositoryProvider`/`MultiRepositoryProvider`, `BlocProvider`/`MultiBlocProvider`, and `DisposableProvider` for controllers.
- For navigation outside widget context, use `NavigationService.navigatorKey.currentState`; the app already wires this key into `MaterialApp`.

## Networking and auth flow

- Use `ApiClient()` as the singleton HTTP client. It wraps Dio with `ApiPath.baseUrl`, long timeouts, curl logging, `TokenInterceptor`, and `ApiTokenInterceptor`.
- `ApiClient.fetch(...)` accepts a path from `ApiPath`, a `RequestMethod`, optional data/rawData/query params/headers/token, retries selected transient network exceptions, converts Dio responses to `RequestResponse`, and displays backend error messages through `ErrorDialogService` for non-200/201 statuses.
- Auth tokens are managed by `AuthService` in `lib/data/services/auth_service.dart`, stored via `SecureStorage`, with `SpUtil` used for user-blocked state. `AuthService.performRefreshToken()` calls `ApiPath.refreshToken`, saves a new access token, and notifies auth binding observers on failures.
- `ApiClient` and `main.dart` currently bypass certificate validation unconditionally; be careful when changing network/security behavior because this affects all requests.

## iOS Fastlane/TestFlight

Run Fastlane commands from `ios/`:

```bash
bundle install
bundle exec fastlane ios build   # build signed IPA only
bundle exec fastlane ios beta    # build IPA and upload to TestFlight
```

Useful non-upload checks from the Fastlane README:

```bash
ruby -c ios/fastlane/Fastfile
ruby -c ios/fastlane/Appfile
plutil -lint ios/Runner/Info.plist
cd ios && bundle exec fastlane lanes
```

Fastlane reads the app version/build from `pubspec.yaml`. Local TestFlight upload expects `ios/fastlane/.env` copied from `.env.example` with App Store Connect key values, especially `APP_STORE_CONNECT_API_KEY_KEY_FILEPATH` pointing to the raw `.p8` file. The GitHub Actions workflow `.github/workflows/ios-testflight.yml` bumps the build number on pushes to `main`, commits it with `[skip ci]`, installs signing assets from secrets, and runs `bundle exec fastlane ios beta` on macOS.

## Tests

Current tests are Flutter tests under `test/`:

- `test/icon_path_test.dart` and `test/image_path_test.dart` verify generated asset path constants reference files that exist.
- `test/widget_test.dart` is the default widget test scaffold.

Use `flutter test <path>` for a single file and `--plain-name` for one named test.
